## [Unreleased]

### Changed

- **`discover --create`'s reachability check is no longer too eager to call a cluster dead.** The per-context probe (`kubectl cluster-info`) used a hard 3-second timeout -- fine for a local or direct cluster, but a real cluster behind a **bastion / IAP tunnel** or a slow/distant endpoint could answer in 4-8s and get **false-skipped** (registered as unreachable and silently dropped). For a "register only what's up" check, a false *unreachable* is the wrong failure direction -- you lose a genuine cluster. The default is now **8 seconds**, and it's tunable via **`STEADYSTATE_REACHABLE_TIMEOUT`** (`3s` to fail faster, `15s` for a far cluster, `0` for no cap). Surfaced by running steadystate against a real GKE-style cluster through an IAP tunnel. Tested: the env override + forgiving default, blank-falls-back-to-default, and that the configured value reaches kubectl. (Also fixed a test-isolation gap where the host's real kube contexts could leak into a "nothing scannable" assertion.)

### Added

- **Natural-language chat -- ask in plain English; the LLM maps it to a vetted command.** When an LLM is configured, a line the typed grammar can't parse is handed to the model, which picks **exactly one verb from the same `tool_schema` the listener dispatches**, fills its args (resolving "the web deployment" to a real fingerprint from the live findings), and hands back a command. The model **parses, it never executes** -- whatever it returns runs through the *exact same* `run_command` + guardrails the typed grammar does. The one safety rule: **effect-tiered confirmation** -- a *read-only* verb (probe/findings/show/...) runs directly; an *effectful* one (approve/decline/fix/run/mute/snooze/send) is **never fired from fuzzy text**, it's echoed back as the concrete command for a human to confirm by sending it. So the model can *suggest* a remediation, but only a human (re)typing the exact command runs it. It's **deterministic-first** (a line that already parses never touches the model -- free, instant, no egress) and off entirely unless an LLM is configured, so chat with no model is byte-for-byte what it was. Rides the analyst's `_complete` seam (kill switch, honest degrade). Validated live against a real k3s cluster: *"restart the web deployment"* -> resolved to the crashlooping workload's fingerprint and offered `run rollout-restart-workload <fp>` for confirmation. Tested: the read-vs-effectful tiering, the vetted-verb guard (a hallucinated verb is dropped), clarify + unmappable handling, an empty/garbage reply degrading to `help`, and the live-state grounding snapshot.

- **`verify` now speaks Helm too -- the other half of real k8s shops.** Verify-the-left landed for Kustomize; this extends it to **Helm charts**, the other dominant way teams declare k8s. **`steadystate verify <chart-dir> --context <cluster>`** auto-detects a chart (`Chart.yaml`) vs an overlay (`kustomization.yaml`), `helm template`s the chart fully client-side (then the same offline kubectl YAML->JSON convert -- no extra tooling beyond helm+kubectl, still no YAML parser), and reconciles it against the live cluster exactly like the Kustomize path. **`--release`/`--values`/`--namespace`** mirror `helm install` -- `--release` matters because chart templates derive resource names from `.Release.Name`, so it must match the installed release for identities to line up (it defaults to the chart dir's name). Both render paths now share one `_RenderedLiveSource` base (render + namespace-scoped reconcile + the coarse presence/image grain), so a third format later is just a `render_*` + a subclass. New **`helm-live`** source, so `scan --source helm-live <chart>` is the same engine; `build_report` gained an opt-in `src=` to accept a pre-built source (a Helm chart needs more than a path -- its release + values). Tested: `render_helm` (release/values/namespace threaded, then the convert), the source's drift, the release-name default, the CLI auto-detecting Helm vs Kustomize and rejecting a dir that's neither, plus the registry + robustness guards.

- **`verify` -- is the running cluster still what Git says? (verify the left).** Drift detection's original promise -- declared vs live -- was hard to *use* in a real environment, because you had to hand-render your declared state to JSON. Your real "left" isn't a flat file; it's a **Kustomize overlay in Git**. So **`steadystate verify <overlay-dir> --context <cluster>`** renders the overlay itself (kubectl's built-in kustomize + a client-side dry-run convert -- **no extra tooling, no YAML wrangling**) and reconciles it against the live cluster, **scoped to the namespaces the overlay touches** so unrelated apps aren't noise. It answers "is prod still what Git says?": a workload **in Git but not running** (ADDED), **running but drifted** from Git -- e.g. an out-of-band image change (MODIFIED), or **running but not in Git** (REMOVED). Coarse on purpose (presence + image + replicas), which is what keeps it *low-false-positive* -- it flags real divergence, not the hundred fields the cluster mutates server-side (the trap that makes most drift tools unusable). This is the runtime-verification layer for shift-left: the left certifies at merge; this is the only thing that knows it **still holds in production**. Backed by a new **`kustomize-live`** source, so it flows through the whole engine -- findings land in the store like any other (muteable, tracked new/recurring/resolved, and within the bound actionable), so you run it *continuously*, not once. `scan --source kustomize-live <dir>` is the same engine; `verify` is the friendly entrypoint. (`run_tool` gained stdin support for the offline render.) Tested: the tolerant kubectl-JSON parse (List / array / single / concatenated), the two-step render, the source's ADDED/MODIFIED/REMOVED with namespace scoping and an all-clear, and the CLI end-to-end.

- **`compliance` -- a CIS benchmark report, grouped by control.** The CIS findings the policy packs already emit were scattered into the normal scan as individual alerts. The new **`steadystate compliance`** command rolls them into a benchmark view: per CIS control, the worst severity and exactly which resources fail it, worst-first, with a header (`CIS Level 1: N controls failing across M resources`). It defaults to **`--source k8s-live`** -- the differentiated case (audit the *running cluster's* posture, not just manifests) -- and also audits a captured snapshot (`--source k8s <file>`) or a compose project. CIS references now carry a **level** (1/2), so `--level` filters (Level 1 by default; `--level 0` for all); `--json` emits the structured report for other tooling. The evaluation runs the packs directly over the resources, so it reports *every* violation, not just ones above the alert-surfacing bar. Honest scope (stated in the command + report): it covers the controls steadystate actually checks and that something violates -- not a full pass/fail over every published control (the control-plane sections need node access and are N/A on managed clusters; a complete catalog with explicit PASS rows is a follow-up). Tested: the grouping (worst-first, deduped resources), the level filter, non-CIS references ignored, the all-clear + JSON shape, and the live CLI path end-to-end (text + `--json`).

- **CIS posture of what's *actually running*, not just your manifests -- the live k8s read now audits security.** steadystate already ships CIS baseline packs (CIS Docker, CIS Kubernetes §5.2 Pod Security), but they only saw the *declared* side -- manifests / captured snapshots. The live cluster read (`kubectl get` for a `k8s-live` target or `probe`) deliberately dropped the security projection, so a running cluster was never CIS-audited. Now the live source projects the pod-security posture (privileged, hostNetwork/PID/IPC, added capabilities, hostPath, allowPrivilegeEscalation, runAsRoot) off the live workloads, so the standing-policy pass flags a **privileged container that's actually deployed right now** -- agentless, straight off the API read, no manifests required. This is the differentiated half of CIS: declared-side scanning is a commodity (Checkov/Trivy/kube-bench in CI); auditing the *live* posture is what steadystate adds. Safe for drift: the live source emits identical declared==observed resources (zero drift) and reconcile strips the posture key anyway, so it can never read as drift. Tested: the live source projects posture (still zero drift), and a privileged live workload surfaces its CIS Kubernetes 5.2.1 finding end-to-end through `scan --source k8s-live`.
- **`findings` takes a keyword now -- grep your findings from chat.** From the CLI you can pipe `findings` to `grep`; in chat you can't, so a long list was hard to filter. Now **`findings <keyword>`** keeps only the findings whose title, fingerprint, severity, or captured evidence (namespace, cluster, the failing pod's last log, ...) contains the keyword -- `findings web`, `findings crashloop`, `findings team-a`. It composes with the status filter and is multi-word: `findings resolved timeout` searches resolved findings for "timeout", `findings all auth error` greps everything. `json` still applies (the filtered list), so an agent can search too. A no-match returns a clear "No findings match '<kw>'", never a silent empty list. One grammar, so it works on every chat surface and the local `chat` REPL at once. Tested: the parser (status word vs keyword, multi-word, json split out), and the filter end-to-end -- title and evidence matches, the json view honoring the keyword, and status+keyword combining.
- **A finding that stops appearing isn't called resolved until it's *stayed* gone -- a grace window.** Errors that repeat in the logs just go away when they're fixed, so a finding clearing is how we learn the problem went away -- but log findings are *sampled*: an error that's still happening can skip a scan or two (the log window rolled, the probe didn't catch it). Resolving on the very first absence made such a finding **flap** -- resolved, then reopened, then resolved -- every other scan. Now a scan only marks a finding resolved once it has been **absent for a grace window** (`now - last_seen >= grace`): sustained lack of signal, not a single gap, is what counts. The window only bites high-frequency probing -- a daily sweep or a CI drift scan is already spaced wider than the grace, so its absences always exceed it and resolution stays immediate as before. Tunable via **`STEADYSTATE_RESOLVE_AFTER`** (`45m`, `2h`, `1d`, a bare number = minutes; `0` restores resolve-on-first-absence); default **30 minutes**. And when a resolved finding **recurs, it reopens keeping its original `first_seen`** -- so you never lose how long it's actually been going (the age is honest across the resolve/reopen cycle; this part was already true and is now covered end-to-end). Tested: the grace holds a finding open within the window and resolves it past it, the env parser (units / bare-minutes / `0` opt-out / junk-falls-back), the full reopen-with-original-first_seen cycle, and the same through the real `reconcile` scan path.
- **The homeostat is visible from chat -- `hold` + `learn`.** The newest value in the product -- the self-maintenance loop (`hold`) and what it's learned (`learn`) -- was reachable only from a terminal. Now two read-only chat verbs surface it, no arguments, no fresh scan (that's `probe`): **`hold`** shows the control-loop posture -- each reflex's earned autonomy (observe/propose/auto) and blast-radius envelope, which of its fixes are *not holding* (the recurrence/self-correction signal), and whether the decider has been granted autonomy (`STEADYSTATE_DECIDER_AUTO`). It answers "what is steadystate maintaining on its own, and is anything not holding?" at a glance. **`learn`** shows what resolved *out-of-band* (a human fixed it, or it self-healed) and what to do -- categories to ADOPT a reflex for, or that SELF-HEAL (mute candidates) -- a suggestion that promotes/mutes nothing. Both are cheap store reads, on every chat surface at once. (A chat `propose` was deliberately left out: an honest "what would the decider do" needs a live scan, which `probe` already triggers -- so chat stays lean.) Tested: both views end-to-end through the dispatch, including the decider-grant reflecting the env.
- **Chat can lift a silence, not just apply one -- `unmute` + `snooze`.** You could `mute` a finding from your phone but had to open a terminal to un-silence it -- the `mute` reply literally told you to go run `steadystate unmute` on the CLI. Now chat has both: **`unmute <fingerprint>`** lifts a mute or snooze (the finding surfaces again next scan; prefix resolution, so a short fingerprint works), and **`snooze <fingerprint> <duration>`** silences it for a while and then lets it return -- `snooze <fp> 2d` / `3h` / `45m` (units h/m/d/w, a bare number is days, matching the CLI's `--days`). An unrecognized duration is reported, never defaulted to a guess; `unmute` only acts on a finding the store remembers. The `mute` reply now points at the chat `unmute`, so the loop stays in chat. Both write through the same store methods the CLI's `unmute`/`snooze` use, on every chat surface at once. Tested: the duration parser (units, bare-days, and junk rejected), both verbs end-to-end through the dispatch, and the parser shapes.

### Changed

- **`discover --create` no longer registers a cluster that isn't actually up.** `discover` builds a `k8s-live` target for every kube context it can see -- but a context lingers in your kubeconfig long after its cluster is gone (a stopped `minikube`, a deleted cluster), so `--create` would dutifully register a dead target you never asked for. Now, on `--create`, each discovered live context is **reachability-checked** -- one fast `kubectl cluster-info --context <ctx> --request-timeout=3s` -- and a context whose API server doesn't answer is **skipped**, reported as `[skipped (unreachable)]`, never written. So a stopped minikube simply doesn't land in `targets.json`. Only `k8s-live` targets are pinged (file/host sources are never contacted), and only on the target-writing path -- the plain `discover` report stays fully offline. **`--no-reachable`** turns it off to register every context even when unreachable (a CI/setup box that can't reach the clusters at discover time). Tested: `context_reachable` targets the right context + carries its kubeconfig + fails fast, and `--create` skips a down context while keeping a reachable one.

- **`discover --create` now overwrites the targets registry instead of merging into it.** Re-running discovery used to *accumulate*: a name already in `targets.json` was skipped, so a cluster/kubeconfig/source that was no longer present lingered forever and a moved path never refreshed -- the registry only ever grew. Now `--create` rewrites the file to exactly what's discovered *now*: a re-discovered target is **refreshed** (e.g. to its current path), an entry that didn't come back is **dropped** and reported as `[removed (stale)]`, and the run still reports each kept one as `[added]`/`[refreshed]`. So a re-run reflects reality -- the fix for stale connections piling up across test runs. (The merge-without-clobber helper it relied on is retired.) Tested: a pre-seeded registry with a stale target and an outdated path, after `--create`, drops the stale one and refreshes the rediscovered one to its live path.

- **Chat stops making you copy 64-char fingerprints -- `approve`/`decline` take a number, a prefix, or nothing.** The heaviest moment on the chat surface was approving a remediation: you had to paste a full 64-hex fingerprint, on your phone, from the `pending` list. Now `pending` **numbers** its rows and `approve <n>` / `decline <n>` act on that number; a bare **`approve`** applies the *only* pending when there's just one (the common case -- no argument at all); and a short fingerprint **prefix** resolves when it's unambiguous. `mute` gains the same prefix resolution while still letting you pre-mute an unseen fingerprint (a `mute-all` correlation key, or known noise). Ambiguity is never guessed -- two matches, or a bare `approve` with several pending, returns a short "say which" instead of acting. This closes an inconsistency too: `show`/`fix`/`send` already accepted prefixes, but the *action* verbs (`approve`/`decline`/`mute`) demanded the exact 64 chars. Applies to every chat surface at once (Slack/Discord/Teams + the local `chat` REPL share one grammar). Tested: the resolver (ordinal / bare-the-only-one / unique-prefix / ambiguous-asks / out-of-range), mute's prefix-with-pre-mute-passthrough, the parser changes, and an end-to-end bare `approve`.

### Added

- **The decider becomes a bounded operator -- `propose --record` puts the LLM's safe proposals in `pending`.** Until now the decider was read-only -- it printed what it *would* do and stopped. Rung 1 of connecting it to the act loop, at the safest setting: **`steadystate propose --record`** runs the decider over findings `hold` has no reflex for, and for each **AUTHORIZED** (within-bound) proposal it **records an approvable pending** -- the model drives *what to do*, a human stays the trigger (`approve`/`fix` runs it through the exact same guardrail + audit). **ESCALATE** (out-of-bound) proposals stay **advisory** (a human initiates break-glass, never the model). **REJECTED** are dropped. So the LLM can only ever populate `pending` with things a human could already have typed -- vetted, bounded, composed from the finding's real keys; it widens nothing (the gate runs *before* anything is recorded). And the proposals are now **grounded**: with `--llm`, the model's prompt includes *how THIS fleet handled the category before* -- the learned lessons (what resolved out-of-band, what self-heals) and the recurrence record (what didn't hold) -- so it reasons from your operational history, not generic k8s. That's the "expert in your fleet" turn: ground, don't fine-tune; the moat is the private memory only your cluster produces. Read-only without `--record`; `--llm` egress-gated with honest degrade to the deterministic `CatalogDecider`. The gate now also honors the active `STEADYSTATE_BOUND` dial (the same consistency `hold`/break-glass have). Pure, tested (`act.decide`: AUTHORIZED→pending / ESCALATE→advisory / REJECTED→dropped; a recorded pending actually runs + audits on `approve`; grounding built from lessons + recurrence and folded into the prompt; the full chat `approve` of a decider-recorded pending). *(This is the suggest rung; the act rung -- granting the decider autonomy within the bound -- follows.)*

- **The decider can act within the bound -- `propose --apply` + `STEADYSTATE_DECIDER_AUTO`.** Autonomy is *granted*, not *earned*. A track record of right answers doesn't retire the hallucination risk the guardrail exists for -- ten green runs leave the next call exactly as fallible -- so the decider serves no probation to "prove" a risk away that it can't. We grant access the way a team grants a new admin's: on day one, scoped to a role, mistakes priced in. The role here is the **bound** (impact × reversibility) + the catalog allow-list -- the guardrail that keeps it *relatively* safe -- and the operator's disaster-recovery plan is the final backstop. So **`STEADYSTATE_DECIDER_AUTO`** is the access grant (the same env-overlay knob that promotes a reflex to auto), and **`steadystate propose --apply`** runs the **AUTHORIZED** (within-bound) proposals straight through the *exact* approve guardrail a human uses -- claim-once, re-validate against the catalog allow-pattern, within-bound re-check, **audited as actor `decider`** so the trail tells an autonomous fix from a human one. The bound stays supreme: **out-of-bound (ESCALATE) proposals never auto-run** under any grant -- they still escalate to a human via break-glass; granting autonomy only moves a within-bound fix from "a human triggers it" to "it runs", never one inch past what you set. Without the grant, `--apply` declines and points you at `--record`. Pure mechanism (`act.act_on_proposals`), tested: the grant reads the env, an AUTHORIZED proposal actually runs and is audited as `decider`, and an out-of-bound proposal never executes regardless of the grant.

- **Break-glass: the out-of-bound, human-only action tier.** The other half of `fix`/`run` -- a human can now deliberately run an action *outside* the autonomous bound (scale a service to zero, delete a node), under friction the system picks for them, while the LLM/code never can. The model is uniform: **one catalog, partitioned by the bound.** Within-bound actions are `fix`/`run` (and, later, the decider). Out-of-bound actions are break-glass -- and *"not anything goes"*: a break-glass action is **still a vetted command shape** (allow-pattern + envelope), so break-glass overrides the **bound** (the *autonomous* limit), **never** the **shape vetting**. An arbitrary command matches no shape and is refused -- `kubectl delete node worker-1234` runs only because there's now a vetted `delete-node` *shape*, not because raw commands are allowed. **The friction scales with the envelope** (`bounds.confirmation_tier`): within bound → no confirmation; out of bound, `recoverable`/`≤ tenant` → a light confirm (`approve <fp>`); `irreversible` **or** `≥ node` → the **strong tier**: `approve <fp>` alone refuses and you must **type the target's name** (`approve <fp> worker-1234`) -- the anti-fat-finger for the most dangerous calls. The run override (`run_catalog_action(break_glass=True)`) skips the within-bound check but still re-validates the allow-pattern; the autonomous callers (`hold`, the decider) never set it, so the bound stays supreme for the code and the model. **Authorization is default-CLOSED**: break-glass is off until `STEADYSTATE_BREAKGLASS_USERS` names the authorized operators (people who'd already have cluster access) -- the signed webhook is the outer boundary, this allowlist the inner who-may-override gate, re-checked at confirm time. Every override is **audited under a distinct `BREAKGLASS` decision**, so `history` shows exactly who crossed the bound, when, and on what. The seam comes seeded with two break-glass shapes -- `scale-to-zero` (light tier) and `delete-node` (strong tier) -- both never auto-offered (no `categories`); you name them. Pure, exhaustively tested (`act.breakglass`/`act.bounds`: the tier math, default-closed auth, the override skipping the bound but not the allow-pattern, and the full chat flow -- challenge → typed-target confirm → run → BREAKGLASS audit, plus the auth re-check at confirm).

### Fixed

- **`steadystate baseline` now works against a cwd kubeconfig.** A discovered target already carries both its `context` *and* its `kubeconfig` (from the cwd-kubeconfig discovery), but the baseline-capture path dropped the kubeconfig -- `baseline <target>` passed only `tgt.context` to `capture_baseline`, which never set it on the live read -- so baselining a cluster reachable only via a cwd kubeconfig failed (kubectl couldn't find that context on the default path). Now `capture_baseline(context, kubeconfig=...)` threads it straight to kubectl (the `KubernetesLiveSource` already knew how -- `use_kubeconfig` from the probe work), and the `baseline` command passes `tgt.kubeconfig`. So you can capture a known-good config baseline for your discovered clusters, and a `k8s-baseline` target reports config drift (a workload added/removed, an image changed) against it -- the positive "capture this good state, tell me what changed" snapshot, which the health-only `k8s-live` path doesn't give. The baseline **file** is now also keyed on the kubeconfig (a short hash folded into the name), so two clusters that share a context *name* across different kubeconfigs -- a common default like `kubernetes-admin@kubernetes` -- get separate baselines instead of clobbering one file (and diffing each against the other's workloads); an ambient-kubeconfig target keeps the old context-only name. Capture and load share the same key. Tested.

### Changed

- **Ansible probe: scale parallelism + timeout to the fleet (fixes the 30s hard-kill on a real inventory).** The probe's 30s budget was per *whole run* -- one `ansible all` process gathering the entire inventory -- and ansible defaults to just **5 forks**, so a 30-40 host fleet ran ~8 serial waves. The heavier hardware (`setup`) disk gather blew past 30s on that and `subprocess.run` **hard-killed the whole run**, losing every host's data (not just the slow ones). Now the probe counts the inventory once (cheap -- `ansible all --list-hosts` does no SSH), cranks **`--forks` toward the fleet size** (capped at 25, so a 40-host run is ~2 waves not 8) and **scales the backstop timeout to the wave count** (≈ a generous budget per parallel wave, floored at 30s) -- so the gather finishes in seconds and the budget fits the fleet. Both are env-overridable for an outlier fleet: `STEADYSTATE_ANSIBLE_FORKS`, `STEADYSTATE_ANSIBLE_TIMEOUT`. One genuinely slow host still can't sink the rest (ansible's own per-host connection timeout drops it), and a failed gather still degrades to no symptoms. The two gathers (service health, disk) share the one sizing. Pure helpers (`_scaled_timeout`/`_forks`) tested for the wave math, the cap, and the env overrides.

- **The kubectl probe no longer flags a pod on restart *count* -- it was noise.** A pod's `restartCount` is **cumulative over its whole life**, so a pod that churned during a deploy weeks ago but is Running and healthy now would trip the old `> 5 restarts` rule and read as a finding -- a false positive an operator burns time confirming is fine. And it added no signal the other way either: a pod that's *genuinely* failing right now is already caught by its bad waiting state (CrashLoopBackOff / ImagePull / …) or a Failed phase -- which is the actual thing you'd investigate; the number "8 restarts" tells you nothing actionable on top. So the restart count is dropped as a detection trigger entirely. It still rides along as **evidence** on a pod that's flagged for a real reason (and as the tiebreaker for which crashlooping pod to name), just never as the reason a pod is surfaced. (`_RESTART_THRESHOLD` removed.)

### Added

- **Issue a vetted fix from chat: `fix <fp>` / `run <action> <fp>` / `actions`.** From the airport, on your phone, you can now *act* on a finding — not just see it. steadystate **offers** a fix per finding (a category → vetted-action map: a crashlooping/erroring workload → `rollout-restart-workload`, an Evicted one → `reclaim-evicted-pods`), and **`fix <fp>`** applies it: it composes the exact command from the finding's stored keys (kind / workload / namespace / cluster-context, plus the kubeconfig resolved from that cluster's target), runs it through the **same approve guardrail** every remediation uses (claim-once, re-validate, argv/no-shell/timeout, **audited under your chat name**), and reports back. A finding with no mapped fix says *"no automated fix — escalate"* rather than guessing. **`run <action> <fp>`** is the cherry-pick form — you name a specific vetted action and the finding supplies the parameters (e.g. `run rollout-restart-workload <fp>`); **`actions`** lists the menu with each action's blast radius. The safety is unchanged and total: every command `fix`/`run` could ever execute must match a **catalog action's allow-pattern** (re-checked at run time) *and* be **within the bound** (an action whose envelope is outside the bound is refused, not run) — so there is no arbitrary-command path; `kubectl delete node ...` matches no vetted shape and is rejected. The generic runner (`act.execute.run_catalog_action`) generalizes the evicted-cleanup executor to the whole catalog, routed through `apply_pending`. Each catalog action now also carries the `categories` it's offered for and a `compose` that builds its command from a finding. **The allow-patterns were widened to accept an optional `--kubeconfig <path>`** (a constrained, injection-tight char class — anchored, no shell metacharacters), so `fix`/`run` reach the cwd-kubeconfig clusters discovery registers. (Operator-initiated and within-bound only; the out-of-bound "break-glass" tier — human-directed higher-risk command *shapes*, never the LLM, with confirmation friction scaled to the envelope — is the next step, its own review.)

- **Ansible host disk-fill check -- the ansible probe now flags a filling filesystem, not just a sick service.** The second classic host malfunction, and the host counterpart to the kubectl probe's node disk %. Alongside the `service_facts` gather, the ansible probe now runs a second read-only gather (the `setup` module's `ansible_mounts`, scoped to `gather_subset=hardware filter=ansible_mounts`) and emits a `DiskFilling` Symptom for any mount **at/over 80% used (MEDIUM), 90% (HIGH)** -- the *same* thresholds as the node disk % check, since a full `/` or `/var` is exactly what *causes* the services the other gather watches to wedge. Each finding is per `host:mount` (`/var is 95% full on web01`), with the percentage + byte sizes in its evidence, so it rides the whole engine: state lifecycle, `mute`, `show`, grouping, `--to servicenow`. The two gathers are independent -- a failed/denied disk read never sinks the service findings, and vice versa -- and the whole probe still degrades to no symptoms when ansible is absent. Pure, tested (`disk_symptoms`/`_mounts_by_host`: the MEDIUM/HIGH split, a healthy mount, unusable size facts skipped, and the probe reporting both axes at once). So `probe ansible-live` (or `scan . --source ansible-live --probe auto`) now reports service health **and** disk fill across the inventory in one pass.

- **`ansible-live` -- live host/service health as a first-class target, the ansible analog of `k8s-live`.** Until now ansible health only ran as a side effect of probing the file-based `ansible` drift source -- which needs a captured `ansible-playbook --check` snapshot -- so there was no way to register a target that just checks live host health against your inventory. `ansible-live` is that: a **pathless** source that reports no drift and exists purely to host the read-only ansible health probe (`ansible all -m service_facts`). **`discover --create` now finds your inventory and registers it automatically.** Because an Ansible inventory has no reliable content signature (INI, YAML, a bare host list, or a dynamic script), discover resolves it the way Ansible itself does: **`ansible.cfg`'s declared `inventory` first** (the authoritative answer, via stdlib `configparser`), then conventional filenames (`inventory`, `inventory.ini/.yml/.yaml`, `hosts`, …), and it **validates each candidate by handing it to `ansible-inventory --list`** (kubectl-style -- the real tool does the parse; when ansible isn't installed it trusts the cfg/name signal). The discovered inventory rides on the target (`Target.inventory`, round-tripped through the targets file) and threads to the probe via `build_report(inventory=...)` -- a new `SupportsInventory` seam parallel to `SupportsContext`/`SupportsKubeconfig` -- so the probe runs `ansible all -m service_facts -i <inventory>` no matter where it runs from. `scan` gained a matching `--inventory` flag. **The payoff with the cwd-kubeconfig discovery: one `discover --create` now covers your whole working directory -- clusters from kubeconfigs *and* hosts from the inventory -- and a single `steadystate sweep` checks them together.** (A probe-only source: its drift path is a deliberate constant `[]` and the live read is the probe's job, which degrades to no symptoms rather than erroring -- so it's exempt from the live-drift-failure guard, explicitly.)

- **`discover` now finds kubeconfigs sitting in your working directory.** A kubeconfig you've dropped in the project dir isn't on kubectl's default path, so `kubectl config get-contexts` never saw it -- `discover` would silently miss a cluster you clearly want probed. Now `discover --create` also looks for kubeconfig files in the cwd (content-sniffed for the `kind: Config` signature, so an arbitrarily-named file like `admin.conf` or `prod-cluster` is still found, and a same-named non-kubeconfig is not), enumerates each one's contexts (handing the file to kubectl via `--kubeconfig`, so kubectl still does the YAML parse -- stdlib-only), and registers a `k8s-live` target per context. Crucially, each target **remembers which kubeconfig it came from**, so a later `probe`/`sweep`/`scan` can actually reach the cluster: `Target` gained a `kubeconfig` field, threaded through `build_report(kubeconfig=...)` (a new `SupportsKubeconfig` seam, parallel to `SupportsContext`) so the live source and the kubectl probe add `--kubeconfig <file>` to every call. Two cwd kubeconfigs that share a context name both register (the file stem disambiguates the target name) -- a real second cluster is never silently dropped. `scan` also gained a matching `--kubeconfig` flag for a one-off read of a context off the default path. (The composed remediation commands a fix suggests -- e.g. the evicted-pod cleanup -- don't yet add `--kubeconfig`; that's a follow-on, since it touches the cleanup allow-pattern.)

- `discover --deep` now **suggests the services a playbook manages** -- an advisory onboarding hint, never a fact the system acts on. It reads the playbook's **roles** (an `haproxy` role -> haproxy is likely on these hosts) and surfaces them, so `--probe ansible` has somewhere to look without the operator configuring from a blank page. Two deliberate boundaries keep it safe: it **rides Ansible's own parser** (`ansible-playbook --list-tasks`, read-only, lists nothing-executed) rather than parsing the playbook YAML ourselves -- the same "ride the tool's output, never raw-parse the file" rule every source follows, and it keeps the core stdlib-only; and it is **purely advisory** -- a human-confirmed suggestion that drives no detection or remediation, so a wrong guess (a custom role name, a dynamic include) just gets ignored, zero blast radius. The role extraction is a pure function tested on fixture output; the live read degrades to no suggestion when there's no playbook, `ansible-playbook` is absent, or the parse fails.

- **Ansible health probe (`--probe ansible`)** -- the malfunction axis for hosts that have no platform health API. Kubernetes/Argo/Helm expose a health verdict steadystate just reads; a bare-metal/VM fleet has nothing equivalent (no `kubectl get` says "haproxy is wedged on web01"). So this reads one with a **read-only** collection we fully control -- the ad-hoc `service_facts` module (no playbook to audit, nothing of the operator's we have to vouch for) -- and turns it into `Symptom`s exactly like the kubectl probe's pod health, so they ride the same reasoning/correlation pipeline. **Observation, not remediation:** a probe never changes anything, so there's no command to gate, no envelope, no bound. The generic, service-agnostic rule (slice 1): a unit in the **failed** systemd state -> a HIGH Symptom; a service that's **enabled** (meant to come up at boot) but **not running** -> MEDIUM (it's supposed to be up and isn't). No need to know *what* the host runs -- operator-declared per-service checks (e.g. keepalived VRRP state) are a later slice. Wired into `--probe auto` for `--source ansible`, so an ansible scan gets drift (`--check`) **and** live host health in one pass. Pure detection rule + callback parser (testable without ansible); the live read shells out once and degrades to no symptoms on any failure (ansible absent, no inventory, unparseable) -- never invents a problem, never breaks a scan. Reads the inventory from `STEADYSTATE_ANSIBLE_INVENTORY`; declared in `steadystate commands` like every probe.
- `rollout-restart-deployment` generalized to **`rollout-restart-workload`**: the catalog action now covers a **StatefulSet** and **DaemonSet** too, not just a Deployment. All three are controllers that recreate their pods with no data loss (a StatefulSet's PVCs persist across a restart), so the `self_healing/service` envelope holds for each -- and the safety stays **pinned by the command shape**: the validator requires `<deployment|statefulset|daemonset>/<name>`, so the action still can never target a bare, non-self-healing pod. Same adversarial vetting (one pinned namespace, no `-A`/`--all`/selectors/extra flags, no injection); a `replicaset/` or other non-controller kind is rejected. (Renames the entry added in the prior release; pre-1.0.)
- Two more vetted **catalog actions** (`act/catalog.py`), growing the safe menu a decider can draw from -- the deliberate way the homeostat's reach expands: one human-vetted, adversarially-tested entry at a time, each with an honest envelope. **`delete-completed-pods`** (Succeeded-phase tombstones in one namespace -- `lossless/tenant`, the cousin of the evicted cleanup). **`rollout-restart-deployment`** (`kubectl rollout restart deployment/<name> -n <ns>` -- `self_healing/service`). The restart's safety is **pinned by the command shape, not inferred from cluster state**: the validator requires the `deployment/<name>` form, so the action can only ever target a Deployment (a controller), so the `self_healing` envelope cannot be applied to a bare, non-self-healing pod -- the allow-pattern *is* the guarantee. Each entry's validator is anchored and adversarially tested: it pins exactly one namespace (never `-A`/`--all-namespaces` -- the one-tenant-vs-whole-fleet firewall), forbids `--all`/selectors/extra flags, and rejects shell-metacharacter injection; both envelopes sit within the default bound (so they may act) and are reachable by the `STEADYSTATE_BOUND` dial (so an operator can narrow them out). The catalog stays the entire trust surface, by design -- so each entry is reviewed like a privileged allow-list line.
- **`steadystate propose` -- an LLM can finally be a *decider* without being a *risk*.** The seam where a model decides *what to do* for a finding `hold` has no reflex for, while a deterministic gate -- blind to who proposed -- decides whether it's allowed. The trick is a **trusted action catalog** (`act/catalog.py`): the source of truth for *(action -> envelope)*. A proposer (a deterministic `CatalogDecider`, or the new `LLMDecider`) may only *name* a vetted catalog action and fill its concrete command; it can **never assert its own envelope**. The gate (`gate_proposal`) then, deterministically and proposer-blind: **REJECTS** anything that isn't a catalog action or whose command fails that action's allow-pattern (anti-injection, re-checked); **ESCALATES** when the action's *catalog* envelope is outside your bound; **AUTHORIZES** the rest. So the worst a hallucinating model can do is name a vetted, bounded action with a command that gets re-validated -- it cannot widen its own blast radius or smuggle an un-vetted command (`nuke-everything` is dropped before the gate even sees it; a `reclaim-evicted-pods` with `; rm -rf /` fails the allow-pattern). The `LLMDecider` rides the existing analyst rails (`LLMAnalyst._complete`): the same egress gate, kill switch, honest no-model degrade, and cost accounting -- so with no model configured it returns nothing and the deterministic decider takes over, never a crash. `steadystate propose` runs it **read-only and advisory** (`--llm` opt-in, egress-gated): it shows what a decider *would* do and the gate's verdict, and **never acts** -- acting on an authorized proposal is a later graduation, the same propose->watch->auto path the reflexes took, so a model's first job is to be right on paper, watched. The catalog is small on purpose (today, the one action steadystate fully possesses -- the evicted-pod cleanup); it grows by *adding vetted actions*, and every decider's reach -- the model's included -- grows with it, always inside the human's bound. Pure, exhaustively tested (`act.decide`: the gate's four verdicts incl. the hallucinated/un-vetted/out-of-bounds cases, both deciders, the LLM constrained to the catalog + degrading honestly, all with a stubbed model -- no network).
- **Action envelopes + the bound -- one autonomy calculus for *all* infrastructure.** The decision an operator makes before any change is always the same two questions -- *how much does this touch, and can I undo it?* -- and it doesn't depend on whether the change is a `kubectl delete`, a `terraform apply`, or an ansible play. So that calculus now lives in one backend-agnostic place (`act/bounds.py`): every action declares an **`Envelope`** -- its **reversibility** (`lossless` → `self-healing` → `recoverable` → `irreversible`) and its **impact** on a generic, cross-backend scale (`ONE` → `SERVICE` → `TENANT` → `NODE` → `FLEET`, each backend mapping its own nouns: a k8s namespace, a terraform workspace, an ansible inventory group, a compose stack all map to `TENANT`) -- and a human declares the **bound**: which envelopes may run unattended. The gate (`within_bounds`) sees only the envelope, never a backend, so **one grid governs every source steadystate can act on**. The default bound is conservative -- only `lossless` (up to a whole tenant) or `self-healing` (up to one service) runs without a human; anything `recoverable`-or-worse, or anything reaching a `NODE`/the `FLEET`, escalates -- and an operator widens it as trust grows (the same graduation `hold`'s reflexes use). This is the spine the autonomy story stands on: a reflex today (and an LLM tomorrow) decides *what to do*; the bound decides *how much it is ever allowed to break*. **The bound is supreme** -- a reflex whose envelope is outside it escalates **even at `auto`**, so flipping a reflex on (or, later, trusting a model) can never cross it; the one decision that never goes to the code or the model is the bound itself. Wired in immediately on both sides of the seam to prove it's universal: the kubectl evicted-cleanup declares itself `lossless/tenant`, and the **terraform** remediation planner (`act/plan.py`) now formalizes its existing prose `blast_radius`/`revert` into the same `Envelope` (a `terraform destroy` is `irreversible` → escalates, mirroring `eligible=False`; an in-place update is `recoverable`) -- so a Terraform apply and a Kubernetes cleanup are judged by the *same* grid. `hold` consults the gate before honoring a reflex's autonomy. Pure, exhaustively tested (`act.bounds`: the gate per reversibility, ordinal impact, an operator-widened policy, and the cross-infra point that terraform + kubectl share the vocabulary; plus `hold` escalating an out-of-bounds reflex even at `auto`).
- **Recurrence-as-confidence -- `hold` becomes self-correcting.** Acting on the known is only safe if the tool also stops when its action *isn't working*. A reflex now earns or loses trust from its own lived track record: a fingerprint steadystate acted on (an `applied`/`verified` remediation in the audit log) that is **open again** is a fix that **didn't hold** -- the problem recurred. `hold` reads that history from the store **before** each tick (per-reflex, keyed by the recurring findings' category) and, once a reflex crosses its **`distrust_after`** budget (default 3), **escalates instead of acting** -- because a fix that won't hold is a *root-cause* problem a cleanup can't reach (evicted pods you keep deleting that keep coming back are a capacity problem, not a cleanup). This is the credit-assignment loop the original vision called for: the **state store is the reward channel**, deterministic and auditable -- no model, no score, just "did our fix actually stick?". It composes with the existing guards (blast-radius, storm) as one more reason to hand off, shows up in the `hold` digest as a normal escalation with its count, and only counts fixes that *didn't* hold (a reflex whose cleanups stay resolved keeps its autonomy). Pure (`reflex_recurrence`), threaded through `plan_hold`/`run_hold`, exhaustively tested (distrust escalates even an in-budget auto reflex; below-budget still acts; only acted-and-open-again counts; the end-to-end `hold --apply` hand-off).
- **`steadystate learn` -- the homeostat learns a response it was never shown.** The control loop (`hold`) acts on *known* responses; this is how the catalog of known responses *grows on its own*. The premise: the human who fixes a fire does it in their own terminal -- we never see the keystroke. So we don't learn from the keystroke; we're a continuous reconciler, so we learn from the **world**: a finding opens, a human acts out-of-band, and on the next probe it's **gone**. The response is latent in that open->resolved transition -- and we can already **attribute** it, because the audit log knows what *we* did. A finding that resolved and is **not** in the audit log's applied/verified set cleared out-of-band (a human, or self-healing) -- a **demonstration**. `learn` groups demonstrations by category and **generalizes** them (anti-unification: the identity dimension that *varies* across them -- namespace, cluster -- becomes a free variable; the category is the constant), then proposes a **lesson**: **ADOPT** -- a category you keep fixing by hand that steadystate already has a reflex for, so *promote it* (`STEADYSTATE_REFLEX_AUTO=<name>`) and `hold` reclaims it next time (closing the loop on knowledge the human keeps re-supplying); or **SELF-HEALS** -- a category nothing of ours answers that keeps clearing on its own (median time-to-resolve shown), a candidate to **mute and stop paging** (the most valuable thing to learn is sometimes to do *nothing*). A lesson's strength is its **count** -- "this would have been the right call N times on findings we already have" -- not a model's confidence, and a lesson is only ever a **proposal**: `learn` is read-only and never promotes a reflex or mutes anything; you do, once you agree. Pure, exhaustively tested (`act.learn`: attribution, the two lessons, the single-demonstration threshold, scope anti-unification, audit-log exclusion). Deliberately coarse today -- the attribution is "did *we* act?", not yet a full world-diff distinguishing a human *delete* from a true *self-heal* -- and honest about it; the richer inference plugs into the same `gather_demonstrations` seam next.
- **`steadystate hold` -- the control loop: stop monitoring, start *holding steady state*.** The first move from detector to homeostat. Most of what goes wrong on a fleet is *known*; the only variance is whether the human who got paged knows the response. `hold` removes that variance for the known cases: it sweeps every target, then for each finding consults a **reflex registry** -- a known stimulus (a Symptom `category`) bound to the autonomy it has *earned*. A reflex that's at `auto` **and within blast-radius** is held **autonomously** (the prober's own safe cleanup, run through the **exact approve guardrail** a human `approve` uses -- claim-once, re-validate against the allow-pattern, execute as argv, **audited under the actor `hold`**); everything else is **escalated**, left pending for a human. The defensible part isn't acting on the known -- it's *declining* to: a single finding over its **per-action** blast-radius (an abnormal pod count) or a **fleet-wide storm** past the per-tick budget (the same finding in N namespaces at once -- a node/capacity problem, not a routine cleanup) is **escalated, never executed**. Reflexes ship **dormant at `propose`**: out of the box `hold` is a **dry run** that prints what it *would* hold and touches nothing -- the mode you watch first. Promote one you've watched be right with `STEADYSTATE_REFLEX_AUTO=<name>` (no code change), then `hold --apply`. Seeded with one reflex we fully possess (`reclaim-evicted`, for Evicted-pod tombstones); the planning (`plan_hold`) is pure and exhaustively tested (act / watch / escalate, blast-radius, storm guard), and a reflex never widens what the tool *can* do -- it only governs whether a human has to be in the loop for something already safe.
- **Proactive node disk warning (`probe --deep`)** -- catch a node *filling up* before it trips `DiskPressure` and starts evicting. `--deep` now reads each node's kubelet **`stats/summary`** (`kubectl get --raw /api/v1/nodes/<node>/proxy/stats/summary`) and flags a node whose worst filesystem (root **or** the image/container fs where logs + images live) is **>= 80% full** -- a `DiskFilling` Symptom at **MEDIUM**, escalating to **HIGH** near full (>= 90%). The early warning the always-on `DiskPressure` condition can't give (that one only fires *at* the eviction threshold). It **skips a node already at `DiskPressure`** (no double-report -- the condition already flagged it, more severely), rides the engine like any Symptom (resolves when freed, `mute`, `show`, `--to servicenow`, `scan --json`), and the identity is context-qualified per node. Opt-in under `--deep` because it's **one `get --raw` per node** and needs `nodes/proxy` RBAC; where that's denied/unsupported it degrades silently to no disk symptoms (the node *conditions* still flag the reactive case). Pure, tested `node_disk_pct`.
- **Node health -- flag a worker node out of disk (or memory/PIDs), or down.** The pod checks miss cluster-*infra* problems; a node whose disk fills up (logs/images) trips the kubelet's **`DiskPressure`** condition, which is what *causes* the pod evictions you then see -- so steadystate now reads each node's `status.conditions` (one `kubectl get nodes -o json`, cheap, always-on) and raises a **HIGH** `Node` Symptom for any node with `DiskPressure` / `MemoryPressure` / `PIDPressure` = True, or `Ready` != True (NotReady/down). It's just another Symptom, so it inherits the whole machinery -- state lifecycle (resolves when the pressure clears), `mute`, `show <fp>`, `--to servicenow`/Slack, and `scan --json` -- and the identity is context-qualified (`<cluster>/Node/<name>`) so the same node name across clusters stays distinct. `kubectl get nodes` is basic cluster-scoped RBAC (far more likely permitted in a locked-down cluster than the `nodes/proxy` metrics endpoint a proactive %-full check would need); a denied/failed read degrades to no node symptoms, never crashing the scan. (A proactive "disk N% full before it's critical" check via the kubelet summary API is the follow-up, for clusters that grant `nodes/proxy`.)
- **Approve-gated evicted-pod cleanup -- the tool can now *run* the fix.** Building on the evicted-pod detection, an evicted finding is recorded as an **approvable remediation** (it shows in `pending`), and `approve <fingerprint>` (chat/CLI) makes steadystate run the `kubectl delete` itself -- through the **same guardrail every drift remediation uses**: claimed once (no double-run, race-safe), recorded to the append-only **audit log**. It's a *command* remediation (a Symptom has no drift executor), so `apply_pending` routes a sentinel `kubectl-cleanup` source to a dedicated runner that **re-validates the command against a strict allow-pattern at run time** and executes it as an argv list (no shell, bounded timeout) -- so the tool can only ever delete pods a human approved, and only the one cleanup shape it generated (an injected/tampered command is refused before anything runs). It **never auto-runs**: even under `--autonomy auto` a cleanup waits for an explicit approve (auto only applies eligible *drift* fingerprints). Honest about reach: a denied/failed delete is reported and audited as FAILED, never raised. Cleanups are offered wherever findings are recorded (a `scan`, a chat `probe`), independent of `--autonomy`.
- **Evicted pods get a category and a fix.** An evicted pod (phase `Failed`, reason `Evicted`) was bucketed as a generic `Failed` (HIGH) with no suggested action. It's now its own **`Evicted`** category at **MEDIUM** (it's a dead tombstone from node memory/disk pressure -- cleanup, not a crash), and the kubectl probe attaches a **concrete recommended action**: `kubectl delete pods -n <ns> --field-selector=status.phase=Failed [--context <ctx>]` -- copy-paste runnable for the right cluster. To carry it, a `Symptom` now has an optional `recommended_action` that the pipeline propagates to its Alert (single and grouped), so every surface -- the chat/console summary, `scan --json`, ServiceNow/Slack -- shows the fix. (Today the tool *offers* the cleanup; making it run the delete behind a guardrail -- approve / a `cleanup` verb -- is the natural follow-up.)
- **`findings` hides resolved by default + status filters.** After a cleanup, the store fills with `resolved` findings that drown out what's still open -- so `findings` (CLI and chat) now **hides resolved by default** and tells you how many it hid. Filter with `findings --open` / `--resolved` / `--muted` / `--all` (CLI) or `findings open|resolved|muted|all` (chat); the default view is everything *except* resolved (open + muted + snoozed). Shared, pure `state.filter_findings` backs both surfaces.
- **`steadystate tools`** -- emit steadystate's chat verbs as a JSON tool/function-call schema, so an LLM agent (e.g. a Microsoft Teams Copilot) can register them as tools and **drive** steadystate, not just read its output. Each tool carries its `args` (with required/optional), its modifier `flags` (for `probe`), and -- crucially -- its **`effect`**: `read-only` (probe/show/findings/…), `state-write` (mute/decline -- reversible, no infra), `guardrailed-write` (approve -- applies a remediation through the executor guardrails), or `external-send` (send -- pushes outward). That guardrail vocabulary lets an agent reason about what's safe to call autonomously vs. what needs a human. Built from the **same `COMMANDS` table** `help`/`chat` render, so the schema can never drift from what the listener actually dispatches (a test pins the coverage). Pure (`inbound.base.tool_schema`).
- **Chat `json` flag** -- a universal modifier that returns a command's result as JSON instead of prose, so an LLM agent (a Teams Copilot driving the chat verbs over the inbound webhook) gets machine-readable output from the *same* verbs a human uses. `show <fp> json` and `findings json` return the stored finding(s) (fingerprint, title, status, severity, first/last-seen, evidence) via a new `serialize.finding_to_dict`; `probe <target> json` (and `probe all json`) return the **same `report_to_dict` shape as `scan --json`**, so there's one contract across CLI and chat. Crucially, **errors come back as JSON too** (`{"error": "..."}`) -- an agent in `json` mode never has to distinguish parseable data from a prose error. Rides the existing flag mechanism (one row in `_FLAG_ALIASES`), and `help` documents it on `probe`/`show`/`findings`.
- **`scan --json` / `probe --json`** -- emit the report as a structured JSON object to stdout instead of the human digest, so a pipeline (or an LLM/agent -- e.g. a Teams Copilot) consumes a scan as data, not scraped prose. One canonical serializer (`serialize.report_to_dict`): a `summary` tally, the `alerts` (worst-first) each with its reasoning (`why`), action + guardrail (`recommended_action`/`remediable`/`remediation_label`), durable keys (`fingerprints`, `correlation_fingerprint`), memory annotations (`status`/`first_seen`), the structured `evidence` a `--deep`/health probe captured, the declared->observed `changes` for drift, and `references`; plus `resolved` (cleared since last scan) and structured `spend`. A superset of the webhook event shape. `scan --json` still reconciles (so status/first_seen/resolved are real) but suppresses the surfaces so stdout is pure, parseable JSON; `probe --json` builds the same report behind the chat `probe` (via a shared `probe_report`) and records findings as usual. The data layer for the agent-facing work (a chat JSON mode + a tool-schema export are the natural follow-ups).
- **`probe --deep` -- log-content health detection.** Until now health was judged from pod *status* (CrashLoopBackOff, ImagePullBackOff, Failed, restart-storm); a pod that's `Running`/`Ready` but **erroring in its logs** (a recovered panic loop, an error flood, an OOM trace) was invisible. `probe --deep` (CLI), `probe <target> deep` / `probe all deep` (chat), and `sweep --deep` add a second pass: for each status-healthy declared workload, read its Running pods' **log tails** (`kubectl logs`, one per pod, capped) and match a curated signature set. **FATAL-class** signatures (`panic:`, segfault/SIGSEGV, OOM, `Traceback`) raise on a single hit -> **HIGH**; the noisier **error** signatures (`ERROR`/`FATAL` log levels, `Exception`, `connection refused`, `context deadline exceeded`, …) must clear a threshold (default 3 lines) -> **MEDIUM**. `ERROR`/`FATAL` are matched case-sensitively (the log-level convention), so prose like "no error occurred" doesn't trip it -- conservative on purpose. It surfaces as an ordinary `Erroring` Symptom (`squid is Erroring in <cluster>/<namespace>`), so it gets the whole machinery for free: state lifecycle (new/recurring/**resolved** when the errors stop), `mute`, `show <fp>` (the matched sample lines), cross-place **grouping** + the group `correlation_fingerprint`, and `send`/`--to servicenow`. **Opt-in** because it costs a `kubectl logs` per pod (the fast path stays one `kubectl get pods -A`); it **skips workloads already flagged by status** (no double-report) and a denied/failed `kubectl logs` stays quiet. Wired through a small `SupportsLogScan` engine seam (`build_report(scan_logs=…)`), mirroring `--context`. Detection lives in a pure, tested `scan_log_text`.
- Chat **`surfaces`** and **`send <fingerprint> <surface>`** -- dispatch a single remembered finding to an alert surface on demand, no full scan. `surfaces` lists every alert target you can send to (`slack`, `teams`, `servicenow`, `pagerduty`, …) and **whether each is configured** (read from the onboarding catalog), so you know what'll actually deliver. `send <fp> servicenow` (or the natural `send <fp> to servicenow`) reconstructs a summary Alert from the stored finding -- its title, severity, captured evidence and timestamps -- and emits it: ad-hoc escalation of "this one matters, file it." It resolves a **unique fingerprint prefix** (like `show`), refuses an unknown or **unconfigured** surface (rather than silently sending nothing), and carries the finding's fingerprint as the Alert's `correlation_fingerprint` so a deduping surface (ServiceNow's `correlation_id`) ties it to the same incident a scheduled `--to` scan would open. A trigger, not a bypass -- it forwards a finding, never remediates. (The reconstructed Alert is a summary: the store keeps the title/severity/evidence, not the original drift/symptom before-and-after.)
- **ServiceNow surface (`--to servicenow`)** -- open (or update) an Incident per Alert via the Table API, so a drift/malfunction that crosses the bar lands as a ticket your existing ITSM process already triages, not just a chat ping. Each Alert maps to one incident keyed by the Alert's fingerprint in ServiceNow's `correlation_id`, so a re-scan **updates the one open incident** (adds a work note) instead of filing a duplicate every run -- the ITSM analog of PagerDuty's `dedup_key`; a **correlated group** rides on its single `correlation_fingerprint`, so the same workload failing in N places is **one** incident, not N. The lookup is scoped to *active* records, so a finding whose incident was resolved/closed and then recurs opens a fresh one; on a failed lookup it conservatively **skips** rather than risk a duplicate. Severity maps to ServiceNow urgency/impact (HIGH/CRITICAL -> 1, MEDIUM -> 2, LOW -> 3); the incident body carries the same reasoning/fix/resources/references the other surfaces show. HTTP Basic auth against `STEADYSTATE_SERVICENOW_INSTANCE` (a bare `dev12345` or a full base URL) + `_USER` + `_PASSWORD`, optional `_TABLE` (default `incident`); registered in the onboarding wizard. Outbound only, stdlib urllib, http(s)-gated; honest degrade when unconfigured. **Closes the loop: when a finding clears, the surface auto-resolves its open incident** (state -> Resolved, with close notes), matched on the same `correlation_id` it opened with -- so a scheduled `scan --to servicenow` closes the tickets it opened, and a grouped finding's incident resolves when the whole group clears. It resolves rather than fully closes -- a human still owns the final close -- and a stateless scan (no `resolved` set) closes nothing.
- A correlated/grouped finding now carries one **`correlation_fingerprint`** -- a single "mute-all" key for the whole group (the same workload failing the same way in N places), **distinct from each member's fingerprint**. Muting it silences the group at once (`mute <correlation-fp>`), without losing the per-instance findings: each error keeps its own fingerprint, still records to the store, and is still inspectable with `show <fp>`. Muting a single member still only quiets that one (the rest of the group keeps firing). The chat probe summary surfaces the key on a grouped alert (`mute-all <fp>  (silences this group)`) above the member fingerprints. The suppression rule is shared by the stateful reconcile and the chat mute-honoring read (`alert_suppressed`): an Alert is withheld when its correlation fp is muted, OR every member is. The key is keyed on `(kind, workload name, category)` -- stable across scans as instances come and go -- and `correlated|`-prefixed so it can never collide with a member's `source|identity|category` fingerprint.
- `show <fingerprint>` (chat) -- **the captured evidence for one finding**: the structured key/value fields a probe recorded (workload, kind, namespace, cluster, unhealthy-pod count, the pod names, max restarts, and the failing pod's **last log line**) plus **when it was first and last seen**, so you can answer "what exactly broke, and was it during the window I care about?". A kubernetes Symptom now carries this structured `evidence` (the kubectl probe fills it); `reconcile`/the summoned-probe record path persist it per fingerprint in a new `findings.details` JSON column (added by the idempotent migration; a re-sighting that carries no evidence preserves what was last captured via COALESCE, so a cheap stateless probe never erases a richer record). The view shows first/last-seen timestamps from the store and accepts a **unique fingerprint prefix** (they're 64 hex chars -- `show 4f72305e` is enough). Each error keeps its own fingerprint, so on a grouped/correlated finding you `show` any one member and get that instance's detail -- one example is enough. Read-only; a finding recorded before evidence capture (or a type that carries none) says so.
- The local `steadystate chat` REPL now has **arrow-key command history** -- up/down through prior commands, plus inline editing (Ctrl-A/E, reverse-search) -- and it **persists across sessions** (saved to `.steadystate/chat_history` next to the state db, on exit -- even on Ctrl-D). It's stdlib `readline`, so no new dependency: on Linux/macOS (the bastion/listener boxes) you get full recall + a remembered history; on Windows, which has no stdlib `readline`, the console already provides its own up-arrow recall, so the helpers are a clean no-op rather than an error. (In a Windows MinTTY shell like Git Bash, which gives neither, run the REPL under a real console -- PowerShell/Windows Terminal, or `winpty steadystate chat` -- for up-arrow.)
- Chat `scan` / `refresh` aliases for `probe` -- "refresh the state" is just re-running a probe (it re-reads live state and records the findings), so the muscle-memory verbs now work: `refresh <target>` / `scan <target>` == `probe <target>`, and a **bare `refresh` / `scan`** refreshes the **whole fleet** (== `probe all`). Resolved in the shared command grammar, so all three providers + the local `chat` REPL get them; `help` documents them on the `probe` line. (Bare `probe` still requires a target -- only the synonyms default to the fleet.
- The same-issue grouping now reaches **across clusters in a `sweep`**: a workload crash-looping in many clusters (a bad image rolled out fleet-wide) collapses into **one correlated finding**, not one per cluster. The sweep regroups the union of every cluster's symptoms by `(kind, workload name, category)` -- each cluster already grouped its own namespaces, so this folds the rest across the landscape -- and the digest gains a `correlated across the fleet (same issue in multiple places)` section listing them (e.g. `squid is CrashLoopBackOff in 12 place(s)`); the same grouped report is what `sweep --to` emits. The per-cluster digest lines are unchanged (you still see *which* clusters are on fire), and reconcile still runs over the un-grouped union so each cluster's new/recurring/resolved memory is exact. The grouping logic is the shared `group_symptom_alerts` (also used in a single scan), and `_place` now distinguishes by cluster+namespace, not just namespace.
- The **same workload failing the same way across namespaces folds into one Alert** -- so a bad image crash-looping every team's `squid` is **one issue to handle, not one per namespace**. The pipeline now groups standalone malfunction Symptoms by `(kind, workload name, category)` -- regardless of namespace -- into a single Alert titled e.g. `squid is CrashLoopBackOff in 12 place(s)`, listing where. (The per-pod case was already one finding: the kubectl probe emits one Symptom *per workload*, aggregating its pods.) Each instance's Symptom rides along, so memory still tracks each (you see which namespaces recover) and the alert carries every fingerprint; it surfaces as one. Mechanical, like the deterministic correlator -- it groups by a shared attribute, it doesn't claim a reasoned root cause -- and order-stable. A Symptom co-located with a Drift still diagnoses into that drift's Alert first (precedence), then the remaining standalone Symptoms group.
- Cluster baseline drift -- `steadystate baseline <target>` + `--source k8s-baseline`: the config-**drift** half of the locked-down-IaC story (the health half is `k8s-live`). For a cluster you have no manifests for, **capture a known-good baseline** (`steadystate baseline prod` snapshots the live workloads to `.steadystate/baseline-<context>.json`), then a `k8s-baseline` target reports **drift against it** -- a workload added/removed since the baseline, or one whose image changed. It reuses the existing k8s reconcile (the baseline is just the *declared* side, live is *observed*), inherits the live read + kubectl health probe from `k8s-live` so **one scan gives both drift and fires**, and qualifies drift identities by context so the shared store stays cluster-distinct. Compared on **presence + images only** -- replicas are dropped, so HPA churn isn't noise. Fixed known-good baseline: capture once, refresh anytime by re-running; with no baseline captured yet it reports no drift (health still works), and a corrupt baseline file is a loud `SourceError`. The live read is cached per scan (one `kubectl get` feeds both the drift reconcile and the health probe).
- `steadystate sweep --to <surfaces>`: a fleet sweep can now **push the fires outward**, not just print a digest. It emits the union of every cluster's alerts (each already labeled with its cluster) to any alert surface (`slack`/`teams`/`webhook`/`pagerduty`/…), so a scheduled bastion/cron sweep pages out with **no inbound endpoint** — one command for the whole fleet instead of a per-cluster `scan` loop. The digest still prints to stdout; `--to` is additive and resolved *before* the slow sweep so a bad surface fails fast. Mutes are honored on the pushed alerts when the sweep runs statefully. `SweepResult` now carries the union `report` for emission.
- Fleet sweep -- `steadystate sweep` (CLI) and `probe all` (chat): probe **every** target in one pass and roll up what's on fire across the fleet (`N cluster(s) -- X on fire, Y clear, Z unreachable`, then a line per cluster, then what recovered since the last sweep). The batch counterpart to the single `scan --target <one>` / `probe <one>`, and the capstone of the fleet-health arc. It's **stateful** (unlike the single, stateless `probe`): each sweep reconciles against the state store, so a cluster that catches fire reads as *new* and one that recovers reads as *resolved* -- "what changed since last sweep," the whole point. The correctness a fleet must get right: a per-target reconcile would call `resolve_absent` with one cluster's fingerprints and so mark every *other* cluster's findings resolved -- so the sweep reconciles **once over the union** of all reports, judging presence fleet-wide; and the live source now **qualifies each identity with its (sanitized) context** so the same workload on two clusters stays two distinct findings (and you can see which cluster). One unreachable cluster is reported inline and never sinks the rest. Local-first (the `chat` REPL + CLI are synchronous); on Slack/Discord `probe all` rides the existing async deferral, and a Teams digest via the outbound webhook is the follow-up for a long fleet.
- `discover --create` now **registers your clusters**: it enumerates the kube contexts visible to kubectl (`kubectl config get-contexts -o name`) and writes one `k8s-live` target per context, so a dir of kubeconfigs (or one config with many contexts) becomes named targets you can `probe` -- the payoff of the live cluster-health path. The target name is a slug of the context (friendly to type in chat), the raw context drives kubectl + labels the alerts, and there's no path (a live source reads live state). kubectl reads/merges the kubeconfig(s) and parses the YAML, so this stays stdlib-only; it degrades to no targets when kubectl is absent or no context resolves (a local kubeconfig read, no cluster contact -- never blocks a create). Folded into the existing `--create` merge, deduped on `(source, path, context)` so live targets sharing an empty path are distinguished by context and a re-run never duplicates; existing entries are never clobbered. So the whole fleet loop is now: `discover --create` (register every cluster) -> `targets` (see them) -> `probe <cluster>` / `scan --target <cluster>` (is it on fire?).
- Named targets can be a **cluster**, not just a file (`Target.context`). A target now carries an optional `context` and its `path` is optional, so a **live** target (a pathless source like `k8s-live`) is just `{"source": "k8s-live", "context": "prod-cluster"}` -- and `scan --target prod` / the chat `probe prod` aim the source *and* the kubectl probe at that one cluster (an explicit `--context` still wins). This is the unit the fleet-health story wants: a target = a cluster, so `targets` lists your clusters and `probe <name>` asks "is this one on fire?". `targets --check` validates a live target by everything *except* its path (reachability is the probe's job at run time, not a static check) and shows the cluster it reaches (`context=…`) in place of a path; the chat `targets` view does the same. The registry stays terse -- `target_to_spec` omits `path` (and `label`/`probe`/`context` at their defaults), so a live target writes just `source` + `context`. The only required field is now `source`.
- Live Kubernetes cluster-health source (`--source k8s-live`): the **"is anything on fire?"** path for a cluster you can reach but have no declared manifests for -- locked-down IaC, a cloud backend, templated repos. It enumerates the cluster's own workloads (`kubectl get deployments,statefulsets,daemonsets -A -o json`) and emits them as **both declared and observed**, so it yields **zero drift by construction** -- its job isn't drift, it's to hand the existing kubectl probe a full list of live workloads to health-check, surfacing the crash-looping / image-pull-failing / restart-storming ones as Symptoms. Reuses the whole engine, so it's **memoryful for free**: run it through `scan` against the state store and the fires get the same new/recurring/**resolved** lifecycle as any finding (a cluster that stops being on fire reads as resolved). It's **pathless** -- `--source k8s-live` needs no positional input (it reads live state itself); the new `--context <ctx>` aims it (and the kubectl probe) at one cluster, so a target = a cluster. Observe-only (never acts); an unreachable cluster surfaces as a loud `SourceError` (exit non-zero), never a false "nothing on fire". Threaded through one small `SupportsContext` seam on the engine, so `--context` reaches whichever of the source/prober opt in and every other source ignores it.
- `discover --json` + `discover --check`: make discovery scriptable. `--json` emits the report as a JSON document (every source/probe `Finding`, the `--deep` `Inspection` facts, and a top-level `scannable` flag) so other tooling consumes it without scraping the human text -- the dataclasses were already frozen, so this is a thin `dataclasses.asdict` serialization. `--check` exits non-zero when nothing is scannable here (no `READY` source and no detected snapshot), a branchable signal for a CI/preflight gate -- the same `--check` idiom `targets --check` already uses, and the default `discover` stays exit-0 (informational), so nothing breaks. The "is there anything to scan here?" verdict is one pure function (`scannable_now`); `--json`/`--check`/`--emit-ci` compose under clear rules (the two output formats are mutually exclusive, and `--create`, which mutates the registry + prints a summary, can't ride a pure-output stream). Also hardened: the hand-maintained `_HINTS`/`_CI_STEPS`/`_REACHABILITY` dicts now have a registry-drift guard test (mirroring the probe registry's `test_auto_keys_are_registered_sources`), so a renamed/added source can't silently lose its hint or CI recipe; and the cwd JSON-snapshot/live-read parsers now also swallow `RecursionError` (deeply-nested JSON), so a pathological file in the cwd can't crash a preflight.
- `discover --emit-ci`: print a **GitHub Actions workflow tailored to the sources discovered here**, instead of hand-editing a generic template. It's the capstone of the discover arc -- `discover` (what's here) -> `--create` (name it) -> `--emit-ci` (run it on a schedule + every PR), the durable home a scan belongs in and the *right* home for the snapshot-file sources (terraform plan / helm list / k8s render) whose backend creds live in CI, not on a laptop. One capture+scan step per discovered source (terraform/helm/docker-compose/k8s/argocd), with the exact capture commands; the **auth** to each backend is left as a clearly-marked `TODO` (OIDC / Vault / kubeconfig vary -- the same honest boundary the hand-authored `deploy/` example draws). A scripting mode: stdout is *only* the workflow YAML, so `discover --emit-ci > .github/workflows/steadystate-drift.yml` just works; progress notes go to stderr, and a cwd with nothing scannable emits nothing. Pure string templating, stdlib-only (no YAML dependency).
- `discover --deep` now inspects **ansible** too, closing the last source-with-a-CLI gap (kubectl/helm/terraform/docker/argocd already covered). It runs the read-only `ansible-inventory --list` and reports the fleet the playbooks target -- host + group counts, the first hosts by name -- then tailors the `--source ansible` capture command to a **real playbook** found in the cwd (`site.yml`/`playbook.yml`/...). Honest about what's safe in a preflight: the `ansible-playbook --check` run that actually yields drift *is* the reconcile (it gathers facts / runs check-mode tasks), too heavy to run unbidden, so deep inspect never runs it -- it surfaces the inventory and hands you the command. Degrades like the rest: `skipped` when ansible isn't installed or no inventory resolves (`ansible.cfg`/`-i`). Playbook detection is filename-based (the engine is stdlib-only -- no YAML parser to confirm a file *is* a playbook). rancher stays advice-only (HTTP/CRD reads, no clean local CLI to interrogate).
- `discover --deep --create` registers **live-discovered compose projects** as targets. The base `--create` only sees the cwd; with `--deep`, the read-only `docker compose ls` sweep finds running projects rooted *elsewhere* and writes a `docker-compose` target per project, pointed at the project's own directory. Honest about the target model's limits: only live-*directory* sources qualify (docker-compose), because a target's path must exist on disk -- argocd/helm/k8s would need a captured snapshot file that isn't there yet, so they stay advice-only (their `--deep` recommendations are unchanged). A compose project *in* the cwd, seen by both passes, is de-duplicated by (source, path) so it's registered once; existing entries are never clobbered. The deep inspection's `Inspection` now carries a structured `targets` tuple -- the machine-readable sibling of its rendered `recommendations` -- which `--create` consumes.
- Microsoft Sentinel enrichment (`--enrich sentinel`): the **security** analog of `--enrich prometheus`. Where Prometheus answers "is this resource unhealthy now?", Sentinel answers "is the SIEM *firing* on it now?" -- a drift that **also** has an active Sentinel incident on its resource (e.g. a firewall opened to `0.0.0.0/0` *and* suspicious access through that exposure) is annotated and **bumped one severity level**, so it pages louder than the same drift on a quiet resource. On-thesis: it reads a verdict the SIEM already computed (an open incident), never runs a detection itself. The operator supplies a **KQL template** keyed by the resource (`{name}`/`{identity}`/...) that returns rows only when there's an active incident -- exactly the PromQL-template contract -- run against the **Log Analytics Query API**. Auth is isolated to the enricher (Azure AD client-credentials: `STEADYSTATE_AZURE_TENANT_ID`/`_CLIENT_ID`/`_CLIENT_SECRET`, token cached per scan), stdlib urllib + http(s)-gated, no Azure SDK. Honest degrade: unconfigured, a failed token, or a flaky Sentinel -> no rows -> no escalation (never escalate on uncertainty), so a SIEM hiccup can't break a scan. `init`/`doctor` cover it.
- Deliver the accept-reality code change (`scan --deliver`): ship a drift's *codify-reality* patch somewhere reviewable -- an axis **orthogonal to `--autonomy`** (which is about enforcing). Two rungs behind one seam (`act/deliver/`, entry-point-discoverable like surfaces): **`patch-file`** writes a `.patch` under `STEADYSTATE_PATCH_DIR` (the auth-free floor -- `git apply` it), and **`github-pr`** opens a PR that codifies the drift via the GitHub REST API, so drift flows through your *normal* review (CODEOWNERS/CI) instead of a steadystate-only approval. The patch stays **deterministic** (no model authors it). **Auth lives only in `github-pr`**: a token from `STEADYSTATE_GITHUB_TOKEN` or the per-workflow Actions `GITHUB_TOKEN` (no personal PAT needed -- the path that works under tight credential controls), passed only in the Authorization header, never logged, http(s)-gated. API-only (blob/tree/commit/ref/PR -- no local `git`, no push credential), idempotent (a stable per-artifact branch reuses the open PR instead of spamming), and honest: an unconfigured adapter, a destructive artifact, or a non-additive diff is skipped, never mis-delivered. `init`/`doctor` cover it.
- Remediation as a reviewable **code change**, folded into `--autonomy suggest`. A drift is `declared != observed`, so it always has two fixes: **enforce** the config (a live apply -- what `suggest`/`approve` already does) or **accept reality** in code (edit the files to match what's there). A suggestion now carries whichever directions exist -- the enforce command *and/or* an **accept-reality patch** -- and `steadystate pending` shows both, so the operator picks at review time. The patch is computed **deterministically** (the model is never in it, exactly as with detection and apply-eligibility). The first (and currently only) renderer covers a Terraform `REMOVED` drift -- a plan `delete`, i.e. config was removed while the resource is *still in state*, so the next apply would **destroy** it. This is precisely the case `suggest` recorded *nothing* for before (it's never apply-eligible). The patch **re-adds the deleted resource block** -- crucially **no `import {}` block** (the resource is already managed; importing an in-state address is a hard terraform error) -- so the declaration exists again and the destroy is averted. It's honest about its limits: observed state can include read-only/computed attributes terraform rejects in a `resource` block, so the rendered block is a reviewed *starting point* (it renders observed scalars, omits nested/computed with a count, never emits `id`, and points at `terraform plan -generate-config-out` to complete it), not a guaranteed plan no-op. A patch-only suggestion is **never** returned for auto-apply, so `auto` still never destroys; and the tool never applies a code change itself -- you `git apply` + merge it, so it works even where automation can't authenticate as a person (e.g. GitHub EMU). Exposed as the optional `Proposer` executor capability (probed by `isinstance`, like the inbound adapters' `defer`/`complete`), so an observe-only or apply-only source simply has nothing to add. Pure renderers (`act/artifact.py`, `act/codify.py`), stdlib only; LLM-authored narrative text is the next increment behind the same `title`/`body` fields, and MODIFIED/ADDED + GitOps (k8s/argocd) renderers are follow-ups.
- Generic webhook + PagerDuty surfaces (`--to webhook` / `--to pagerduty`). Both ride the existing Surface seam, so they're a registry line each and unlock the ITSM/paging world without building a native integration per vendor. **`webhook`** POSTs each alert as a provider-agnostic JSON event (`title`/`severity`/`why_it_matters`/`recommended_action`/`resources`/`references`/`fingerprints`/...) to `STEADYSTATE_WEBHOOK_URL` -- point it at Opsgenie/Jira/ServiceNow (or a small middleware) and route from there. **`pagerduty`** opens an incident per alert via the Events API v2 (`STEADYSTATE_PAGERDUTY_ROUTING_KEY`), keyed by the alert's fingerprint as PagerDuty's `dedup_key` so re-scanning the same drift updates the one open incident instead of spamming -- fitting the "page louder" story (a drift breaching a PromQL bar *right now* should open an incident, not just post to a channel); steadystate severity maps to PD `info/warning/error/critical`. Both are outbound-only, stdlib urllib, http(s)-gated, and degrade honestly when unconfigured; `init`/`doctor` cover them.
- `scan --confirm-llm`: review every model prompt before it's sent -- an opt-in egress gate for cautious operators. Before any prompt leaves the box, it prints the destination (provider + model) and the *exact* prompt (the instruction + your declared/observed state), then asks; decline and nothing is sent -- the scan degrades to deterministic, the same honest path as `--no-llm`. It's a per-call data-egress review and, incidentally, a hard spend gate (no paid call without a yes). Opt-in and **off by default**, so headless scans, the CronJob, and the listener are unchanged. Interactive only: with no terminal to ask on it fails *closed* (runs without the LLM rather than blocking). Implemented as an optional gate on the analyst (`LLMAnalyst(gate=...)`, threaded through `engine.build_report`); no capability or config data is added to the prompt, so what you approve is all that goes out.
- Honest remediations: each alert says whether steadystate can actually carry out the fix, decided deterministically -- never by the LLM. The model proposes a fix in prose; a separate layer (`engine._stamp_remediability`) stamps each alert `remediable` from the **same executor + apply-eligibility the suggest/approve path uses**, so a surface labels the recommendation `steadystate can apply this -- approve to reconcile.` or `Manual -- outside what steadystate executes.` (rendered on console / Slack / Teams + the chat probe's verbose evidence). This closes the gap where a recommendation looked actionable but had no Approve behind it -- e.g. a pod crashloop (a Symptom, no executor) now reads as manual, while a terraform drift reads as applyable. The LLM is also **prompt-hardened** to never speculate about what steadystate can or cannot do, or whether a command is "allowed" -- that verdict is the tool's, computed deterministically. No capability data is sent to the model (nothing extra leaves the box).
- Probes declare their commands (`kubectl logs` is now first-class). The kubectl/docker health probes already shell out to `kubectl logs` / `docker logs` for a failing pod's evidence -- but those commands weren't in any permission manifest, so `steadystate commands` understated what the tool runs and a least-privilege RBAC built from it would miss `pods/log`. Probes now carry an observe-only `Capabilities` manifest (mirroring sources), surfaced in `steadystate commands` (a new `<probe> (probe)` section) and the catalog's Probes seam (with observe counts). `PROBE_CAPABILITIES` declares: kubectl -> `kubectl get pods -o json` + `kubectl logs --tail --previous`; docker -> `docker ps --format json` + `docker logs --tail`; argocd -> none (reads a captured snapshot). The deploy RBAC notes that tightening from `*` to an explicit list needs both `pods` and the `pods/log` subresource for the probe. Transparency + correct RBAC; no behavior change to what the probe collects.
- Setup wizard + preflight (`steadystate init` / `steadystate doctor`): onboarding without reading the source. `init` is an interactive wizard that walks the capabilities -- LLM reasoning, alert surfaces, chat-back listener secrets, source credentials, enrichment -- prompts only for what you choose (secret input hidden), and writes a gitignored `.env` (chmod 600 on POSIX), merging into an existing one without wiping untouched keys. `doctor` is the read-only counterpart: it inspects the live environment (plus an optional `--env-file`) and reports every capability as `ready` / `partial` / `off` with what each still needs -- reporting *whether* a var is set, never its value, so it's safe to run and paste. Both read one declarative catalog (`onboarding.py`) of the ~25 env vars the engine actually consumes, so the config surface lives in exactly one place. Built on the typer/rich CLI already in use; the engine stays stdlib-only.
- Out-of-tree plugin discovery (importlib entry points): the registries stop being repo-bound. Every seam -- sources, domains, surfaces, inbound adapters, executors, correlators -- now overlays `importlib.metadata` entry points (`steadystate.<seam>`) on its built-ins, so a *separately installed* package extends steadystate without forking it: declare `[project.entry-points."steadystate.sources"]` (or `.domains`, `.surfaces`, …) pointing at the same shape the in-tree registry holds (a source factory, a `Surface` factory, a `Domain` class). Discovery is **isolated** -- a plugin that fails to import is logged and skipped, never crashing the host or hiding the plugins that load -- and **safe**: built-ins win every name clash, so installing a package can *add* `--source pulumi` but never silently redirect `--source terraform` at its own code. A discovered source factory that advertises a `commands` manifest also shows up in `steadystate commands`. Stdlib only (`plugins.py`, no plugin framework); this closes the last in-tree-only seam, so "add a pack, never edit core" now holds for third parties too.
- Helm source (`--source helm`): rides a release's own status. Helm tracks each deployment as a *release* whose healthy status is `deployed`; one stuck in `failed` / `pending-upgrade` / `pending-rollback` / `pending-install` or mid-`uninstalling` is almost always an interrupted or broken `helm upgrade` -- a class of malfunction the resource-level `k8s` source doesn't see. It reads `helm list --output json` (core Helm, no plugin), turns every not-`deployed` release into a Drift (`pending-install → added`, `uninstalling → removed`, else `modified`), and carries the chart + revision as observed context. Observe-only by declaration (`helm list/status/get manifest`); `helm upgrade/rollback/uninstall` are flagged destructive. Per-resource config drift (deployed manifests vs the chart, via helm-diff) is a follow-up. One registry line, like every source.
- Async summons: a slow `probe` no longer times out in chat. A live `terraform plan` can take longer than a chat provider's ~3s interaction window, so for `probe` the listener now ACKs immediately and runs the scan in a background thread, posting the result back when it's ready -- Discord edits its deferred ("thinking...") message, Slack POSTs the slash command's `response_url`. It's an optional `defer`/`complete` adapter capability (probed by attribute, like the rest of the seam), so Teams -- which has no `response_url` -- stays synchronous and unchanged; `dispatch` returns the deferred work alongside the immediate reply and the handler runs it off the request path. The post is read-only + best-effort: a flaky callback never crashes the listener. Fast verbs (help/pending/cost/...) still answer synchronously.
- `scan --verbose` / `-v`: render the **evidence** on the console for a full scan -- the `declared → observed` before/after per drift (plus a policy finding's / symptom's specifics), so a scheduled scan can be *audited*, not just trusted. The reasoning, framework references, and recommended action already rendered; this adds the raw diff the terse output dropped. Completes the reviewability track started with `probe <target> verbose` (chat + CLI), now covering the main scan path too. Console-only and off by default, so the normal output is unchanged.

### Fixed

- A Kubernetes malfunction now names **where** it's firing -- the namespace, and the cluster too on a fleet. The symptom title was `squid is CrashLoopBackOff` with no location, so across many clusters/namespaces you couldn't tell *which* squid. The title is the one field every surface shows (the chat `probe`/`scan` summary, the scan console panel, and the remembered `findings` row -- which stores only the title), so the location goes there: `squid is CrashLoopBackOff in <context>/<namespace>` on a fleet (a target = a cluster), or `... in <namespace>` on a single ambient cluster. Grouping is unaffected (`group_symptom_alerts` keys on the identity-derived workload name, not the title), so the same workload still folds across namespaces/clusters into one `... in N place(s)` alert.
- Chat `probe all` now reads like a single `probe`: each fleet finding shows its **description and fingerprint(s)**, not just a tally and a bare title -- so the sweep says *what* is wrong and stays `mute`able. And `verbose` finally does something on it: `probe all` and `probe all verbose` were identical (the sweep hard-coded `verbose=True`), so the flag was dead and you got `squid is CrashLoopBackOff in 2 place(s)` with no detail underneath. Now the reply is the cluster tally followed by the deduped fleet findings rendered through the **same** summary the single probe uses (description + fix + fingerprints by default; full per-symptom evidence under `verbose`). The CLI `sweep` digest is unchanged (it keeps its terse "correlated across the fleet" roll-up); the detail is added only on the chat path, which is where you act on findings.

- A chat/CLI `probe` reply now shows **what's wrong**, not just a title and fingerprints. Each alert prints its one-line description (`why_it_matters` -- e.g. the failing pod's evidence, or `2 instances of squid are CrashLoopBackOff across: …`) and its recommended fix (when there is one) by default; `verbose` still swaps that for the full declared->observed evidence. Previously the non-verbose summary listed only the severity + title + `fp <hash>` lines, so an operator couldn't tell what the finding *was* without re-running with `verbose`.

- A pod deleted *between* the kubectl probe's `get pods` and its log fetch (or any pod with no `--previous` container) no longer logs a scary **WARNING** for the routine `kubectl logs` failure. The probe always handled it (the symptom still surfaces, just without a log tail), but the failed `--previous` attempt -- which is expected on most pods -- warned every time, surfacing as noise (e.g. a JSON log line under a structured logger). A failed *logs* fetch is now best-effort/debug; a failed pod *list* (which means no symptoms surfaced) still warns. Pinned with a test: a crash-looping pod deleted mid-probe still produces its symptom and emits no warning.

- The local `chat` REPL (and the listener) now resolve targets from **`./steadystate.targets.json`** when `STEADYSTATE_TARGETS` isn't set -- the same default `scan --target` / `targets` use -- so a `discover --create` registry is picked up without exporting an env var (it wasn't before: `chat` saw no targets unless you set the env). The default registry filename is now **`steadystate.targets.json`** (was a bare `targets.json`), so discovery never reads or clobbers an unrelated `targets.json` that happens to be in the cwd. Also: the chat `help` now documents **`probe all`** (the fleet sweep) on the `probe` line -- it was reachable but undocumented -- and the "no targets configured" replies now point at `discover --create`.
- The kubectl probe now attributes a pod to its workload by the pod's controller **`ownerReference`** (the ReplicaSet for a Deployment, the StatefulSet/DaemonSet directly), not a name-prefix guess. The old `name.startswith("<workload>-")` over-matched when two workloads in one namespace had overlapping names -- `squid` would have claimed `squid-proxy`'s pods (its pod names start with `squid-` too), so a healthy `squid-proxy` could read as a crash-looping `squid`, or one's failure get mis-attributed to the other. Now a Deployment matches only its ReplicaSet `<workload>-<pod-template-hash>` (the hash is a single dash-free segment, so `squid` no longer matches `squid-proxy`'s `proxy-<hash>` suffix), and a StatefulSet/DaemonSet matches its controller name exactly. A bare pod with no controller falls back to the legacy name match, so nothing else changes. (Same-name-across-*namespaces*, e.g. `squid` in ten namespaces, was already correct -- the namespace is part of the resource identity at every layer; this fixes the same-name-in-one-*namespace* edge.)
- Shared SQLite store is now concurrency-safe (it's shared by design -- a scheduled scan and the chat listener, which writes from per-command daemon threads, hit one file). The connection opened with a bare `sqlite3.connect` -- no WAL, no busy timeout -- so a concurrent write could throw `database is locked` and drop an approval or crash a scan. It now opens in **WAL** (readers + the single writer don't block each other) with **`busy_timeout=5000`** (a contended write waits-and-retries instead of erroring) and `synchronous=NORMAL`. Also closed a **TOCTOU in `apply_pending`**: two chat users approving the same fingerprint both read it as `pending` and both ran the remediation. Approval now goes through a new atomic `claim_pending` (a conditional `UPDATE ... WHERE status='pending'`), so exactly one approver wins and the irreversible remediation runs **at most once** -- verified with an 8-thread, separate-connection race test.
- `--source k8s --probe auto` now actually runs the kubectl probe. The auto-probe map keyed on `"kubernetes"` while the Kubernetes source registers as `"k8s"`, so `auto_prober_for("k8s")` returned `None` -- a default-config scheduled K8s scan silently surfaced no CrashLoopBackOff / failing-pod malfunctions (the whole point of the probe), unless an operator knew to pass `--probe kubectl`. docker-compose and argocd matched their source names; K8s was the only broken one. It shipped green because two tests asserted against `"kubernetes"` -- the tests encoded the bug. Fixed the key, corrected the tests, and added a guard (`test_auto_keys_are_registered_sources`) that fails if any auto-probe key isn't a registered `--source`, so this class of mismatch can't ship silently again.
- Drift sources no longer crash the scan (or hang forever) on infra-tooling hiccups. The live `_run_*` methods shelled out with `check=True` and **no timeout**, and `scan` only caught `ValueError` -- so a missing binary (`FileNotFoundError`), a non-zero exit (`CalledProcessError`), or empty/garbage output (`JSONDecodeError`) surfaced as a raw traceback, and a hung `terraform plan` / `kubectl` blocked indefinitely. All five live sources (terraform · k8s · docker-compose · ansible · helm) now route through a shared `run_tool` + `loads_json` (`sources/base.py`): a hard, per-source `timeout` (terraform/ansible 300s, k8s/compose/helm shorter -- all configurable) and uniform conversion of every failure into a clean **`SourceError`**. `scan` catches it, prints `scan failed: <reason>` and exits non-zero; the listener already wrapped it. Crucially the source **raises** rather than returning empty -- a drift scan going silent would be a false "no drift" all-clear, worse than a loud failure (the asymmetry with the best-effort symptom probe, which *may* degrade to empty, is now deliberate and documented).
- The HTTP-fetching sources get the same treatment. ArgoCD and Rancher called `safe_urlopen` with **no timeout** (block forever) and caught no HTTP/JSON errors, so a 401/500 or a hung server took the scan down with a raw traceback. Both now fetch through a shared **`fetch_json`** (`sources/base.py`, the urllib parallel to `run_tool`): a bounded `timeout` (30s, configurable) and every failure -- an HTTP error (with the status code), an unreachable/hung server, or an unparseable body -- converted to a clean `SourceError`, handled exactly like the subprocess sources above.
- `fix` now reports a live-tool failure cleanly, like `scan`. `fix` collects drift through the same `collect_drift()` path, but -- unlike `scan` -- didn't catch the `SourceError` the above fixes introduced, so a missing binary / non-zero exit / timeout during `steadystate fix` still surfaced as a raw traceback. It now catches it, prints `fix failed: <reason>` and exits non-zero, closing the last spot the M1/M2 hardening hadn't reached on the sibling command.

### Changed

- One autonomy gate for all of `--autonomy auto`: the **bound now governs the drift apply path too**, not just reflexes/`hold`. Previously two encodings of "safe to act" coexisted and disagreed -- a drift's `eligible` boolean gated terraform/ansible auto-apply, while the `Envelope`/`within_bounds` bound (from #147) gated only reflexes. The two are now layered into one model: `eligible` answers *"may a human approve this?"* (still never a destroy), and a new `can_run_unattended` (act/plan.py) adds the stricter *"may it run unattended?"* -- which requires the action's envelope to sit within the bound. **Behavior change:** a terraform `ADDED`/`MODIFIED` drift is `recoverable`, which the conservative default bound does not allow unattended, so `--autonomy auto` now **holds it for approval** (recorded as pending, reported on the run) instead of auto-applying -- a recoverable infra change no longer applies with no human in the loop by default. An eligible-but-out-of-bound drift is recorded for `approve` exactly as before; only the *unattended* application is gated. (An executor that hasn't declared envelopes yet -- ansible -- falls back to `eligible` alone, so its behavior is unchanged; the migration is incremental.)
- New **`STEADYSTATE_BOUND`** dial widens/narrows the autonomous bound without code (the same env-overlay idiom `STEADYSTATE_REFLEX_AUTO` uses), and **every** autonomy path -- drift auto-apply, reflexes, and the decider -- honors it, so one knob governs the whole tool. Comma-separated `reversibility=impact` pairs name the highest impact tier allowed unattended per reversibility (`none` forbids it); e.g. `STEADYSTATE_BOUND="recoverable=service"` re-enables auto for a single-service recoverable change (the dial a terraform operator turns after watching `auto` be right). Unparseable tokens are skipped, never applied -- a typo can only leave the bound at the conservative default, never silently widen it. The `bounds.py` "one grid governs every source" claim is dialed back in-code to match what's actually enforced today (the autonomous paths + any executor that declares an envelope; un-migrated executors fall back to `eligible`).
- The default targets registry moved from `./steadystate.targets.json` to **`.steadystate/targets.json`** -- under the same gitignored state dir as `state.db`, instead of a loose file in the cwd. `discover --create` writes it there (creating `.steadystate/` as needed) and the chat/CLI fallback reads it there; an explicit `STEADYSTATE_TARGETS` still wins, and a foreign `./targets.json` is still never read. If you have a `steadystate.targets.json` from before, re-run `discover --create` (writes the new path) or move it under `.steadystate/`.

- The kubectl probe fetches pods in **one `kubectl get pods -A`** (grouped by namespace client-side) instead of one `kubectl get pods -n <ns>` per namespace. Across a fleet with many namespaces that's 1 API round-trip instead of N -- a real speedup for the live cluster-health path / `sweep`. It also short-circuits the read entirely when the scan has no kubernetes resource (a non-k8s scan never shells out to kubectl). Same RBAC as before (cluster-wide pod read; the built-in `view` ClusterRole covers it), and the per-pod `kubectl logs` for evidence is unchanged.

- A summoned **`probe <target>`** (CLI + chat) now **records its findings** to the state db (and creates the db if absent), so they show in `findings`, get new/recurring memory, and can be `mute`d -- previously a probe persisted nothing, which surprised operators ("I probed, saw findings, but no state file"). It's **record-only**: it never calls `resolve_absent`, because a single probe isn't a full-fleet view and must not mark *another* target's findings resolved (that scoping is exactly why `sweep` reconciles over the union). It still never *applies* -- chat stays a trigger, not a bypass; only the apply path is gated, and recording observed findings was never the bypass. Shared `seen_findings` helper between `reconcile` and the probe's record-only path.

- Remediation verify (terraform): the post-apply re-check now counts only *actionable* residual drift and retries briefly. After a clean apply, a full `terraform plan` often lists the resource under refresh-only `resource_drift` (server-set fields / cloud read-after-write lag) while it is a no-op in `resource_changes` -- terraform has nothing left to apply. The verify previously read that as "applied, but not verified"; it now correctly reports **verified** (validated end-to-end against live GCP), while a genuinely unreconciled drift (e.g. a provider that merged a `TypeSet`) still honestly reports not-verified.
- Security exposure checks for set-typed attributes (CIDR ranges, IAM member lists) key off the observed/reality side rather than a declared-vs-observed diff, so terraform's union-encoding of a `TypeSet`'s planned value can't hide a real open-to-world rule.
- Version is single-sourced from `__version__` in `src/steadystate/__init__.py` via `[tool.hatch.version]`.
- Security match uses word-boundary kind matching instead of loose substring (avoids the over-match trap).
